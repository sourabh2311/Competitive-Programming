    We are given a function f(x)f(x) which is unimodal on an interval [l;r][l;r]. By unimodal function, we mean one of two behaviors of the function:

    1. The function strictly increases first, reaches a maximum (at one point or at a segment), and then strictly decreases.

    2. The function strictly decreases first, reaches a minimum, and then strictly increases.

    In this article we will assume the first scenario, and the second is be completely symmetrical to it.

    The task is to find the maximum of function f(x)f(x) on the interval [l;r][l;r].

    Algorithm

    Consider any 2 points m1m1, and m2m2 in this interval: l<m1<m2<rl<m1<m2<r. We evaluate the function at m1m1 and m2m2, i.e. find the values of f(m1)f(m1) and f(m2)f(m2). Now, we get one of three options:

    • f(m1)<f(m2)f(m1)<f(m2)

      The desired maximum can not be located on the left side of m1m1, i.e. on the interval [l;m1][l;m1], since either both points m1m1 and m2m2 or just m1m1 belong to the area where the function increases. In either case, this means that we have to search for the maximum in the segment [m1,r][m1,r].

    • f(m1)>f(m2)f(m1)>f(m2)

      This situation is symmetrical to the previous one: the maximum can not be located on the right side of m2m2, i.e. on the interval [m2;r][m2;r], and the search space is reduced to the segment [l;m2][l;m2].

    • f(m1)=f(m2)f(m1)=f(m2)

      We can see that either both of these points belong to the area where the value of the function is maximized, or m1m1 is in the area of increasing values and m2m2 is in the area of descending values (here we used the strictness of function increasing/decreasing). Thus, the search space is reduced to [m1;m2][m1;m2]. To simplify the code, this case can be combined with any of the previous cases.

    Thus, based on the comparison of the values in the two inner points, we can replace the current interval [l;r][l;r] with a new, shorter interval [l′;r′][l′;r′]. Repeatedly applying the described procedure to the interval, we can get arbitrarily short interval. Eventually its length will be less than a certain pre-defined constant (accuracy), and the process can be stopped. This is a numerical method, so we can assume that after that the function reaches its maximum at all points of the last interval [l;r][l;r]. Without loss of generality, we can take f(l)f(l) as the return value.

    We didn't impose any restrictions on the choice of points m1m1 and m2m2. This choice will define the convergence rate and the accuracy of the implementation. The most common way is to choose the points so that they divide the interval [l;r][l;r] into three equal parts. Thus, we have

    m1=l+(r−l)3m1=l+(r−l)3

    m2=r−(r−l)3m2=r−(r−l)3

    If m1m1 and m2m2 are chosen to be closer to each other, the convergence rate will increase slightly.

    Run time analysis

    T(n)=T(2n/3)+1=Θ(logn)T(n)=T(2n/3)+1=Θ(log⁡n)

    It can be visualized as follows: every time after evaluating the function at points m1m1 and m2m2, we are essentially ignoring about one third of the interval, either the left or right one. Thus the size of the search space is 2n/32n/3 of the original one.

    Applying Master's Theorem, we get the desired complexity estimate.

    The case of the integer arguments

    If f(x)f(x) takes integer parameter, the interval [l;r][l;r] becomes discrete. Since we did not impose any restrictions on the choice of points m1m1 and m2m2, the correctness of the algorithm is not affected. m1m1 and m2m2 can still be chosen to divide [l;r][l;r] into 3 approximately equal parts.

    The difference occurs in the stopping criterion of the algorithm. Ternary search will have to stop when (r−l)<3(r−l)<3, because in that case we can no longer select m1m1 and m2m2 to be different from each other as well as from ll and rr, and this can cause infinite iterating. Once (r−l)<3(r−l)<3, the remaining pool of candidate points (l,l+1,…,r)(l,l+1,…,r) needs to be checked to find the point which produces the maximum value f(x)f(x).

    Implementation

    double ternary_search(double l, double r) {
    double eps = 1e-9;              //set the error limit here
    while (r - l > eps) {
        double m1 = l + (r - l) / 3;
        double m2 = r - (r - l) / 3;
        double f1 = f(m1);      //evaluates the function at m1
        double f2 = f(m2);      //evaluates the function at m2
        if (f1 < f2)
            l = m1;
        else
            r = m2;
    }
    return f(l);                    //return the maximum of f(x) in [l, r]
}

    Here eps is in fact the absolute error (not taking into account errors due to the inaccurate calculation of the function).

    Instead of the criterion r - l > eps, we can select a constant number of iterations as a stopping criterion. The number of iterations should be chosen to ensure the required accuracy. Typically, in most programming challenges the error limit is 10−610−6 and thus 200 - 300 iterations are sufficient. Also, the number of iterations doesn't depend on the values of ll and rr, so the number of iterations sets required relative error.
