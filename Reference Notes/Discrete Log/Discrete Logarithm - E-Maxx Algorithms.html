    The discrete logarithm is an integer xx solving the equation

    ax≡b(modm)ax≡b(modm)

    where aa and mm are relatively prime. (Note: if they are not relatively prime, then the algorithm described below is incorret, though it can be modified so that it can work).

    In this article, we describe the Baby step - giant step algorithm, proposed by Shanks in 1971, which has complexity O(m−−√logm)O(mlog⁡m). This algorithm is also known as meet-in-the-middle, because of it uses technique separation of tasks in half.

    Algorithm

    Consider the equation:

    ax≡b(modm)ax≡b(modm)

    where aa and mm are relatively prime.

    Let x=np−qx=np−q, where nn is some pre-selected constant (we will describe how to select nn later). pp is known as giant step, since increasing it by one increases xx by nn. Similarly, qq is known as baby step.

    Obviously, any value of xx in the interval [0;m)[0;m) can be represented in this form, where p∈[1;⌈mn⌉]p∈[1;⌈mn⌉] and q∈[0;n]q∈[0;n].

    Then, the equation becomes:

    anp−q≡b(modm)anp−q≡b(modm).

    Using the fact that aa and mm are relatively prime, we obtain:

    anp≡baq(modm)anp≡baq(modm)

    This new equation can be rewriten in a simplified form:

    f1(p)=f2(q)f1(p)=f2(q).

    This problem can be solved using the method meet-in-the-middle as follows:

    • We calculate f1f1 for all possible values of pp. Sort these values.
    • For each value of qq, calculate f2f2, and look for the corresponding value of pp using the sorted array of f1f1 using binary search.

    Complexity

    For each value of pp, we can calculate f1(p)f1(p) in O(logm)O(log⁡m) using binary exponentation algorithm. Similar for f2(q)f2(q).

    In the first step of the algorithm, we need to calculate f1f1 for every possible values of pp, and then sort them. Thus, this step has complexity:

    O(⌈mn⌉(logm+log⌈mn⌉))=O(⌈mn⌉logm)O(⌈mn⌉(log⁡m+log⁡⌈mn⌉))=O(⌈mn⌉log⁡m)

    In the second step of the algorithm, we need to calculate f2(q)f2(q) for each possible value of qq, and then do a binary search on the array of values of f1f1, thus this step has complexity:

    O(n(logm+logmn))=O(nlogm)O(n(log⁡m+log⁡mn))=O(nlog⁡m).

    Now, when we add these two complexity, we would get logmlog⁡m multiplied by nn and m/nm/n, which has minimum value when n=m/nn=m/n, which means, to achieve optimal performance, nn should be chosen such that:

    n=m−−√n=m.

    Then, the complexity of the algorithm becomes:

    O(m−−√logm)O(mlog⁡m).

    Implementation

    The simplest implementation

    In the following code, function powmod performs binary exponential ab(modm)ab(modm), and function solve produces a proper solution to the problem. It will returns −1−1 if there is no solution, and returns one possible solution in case a solution exists.

    int powmod (int a, int b, int m) {
    int res = 1;
    while (b > 0)
        if (b & 1) {
            res = (res * a) % m;
            --b;
        }
        else {
            a = (a * a) % m;
            b >>= 1;
        }
    return res % m;
}

int solve (int a, int b, int m) {
    int n = (int) sqrt (m + .0) + 1;
    map<int,int> vals;
    for (int i=n; i>=1; --i)
        vals[ powmod (a, i * n, m) ] = i;
    for (int i=0; i<=n; ++i) {
        int cur = (powmod (a, i, m) * b) % m;
        if (vals.count(cur)) {
            int ans = vals[cur] * n - i;
            if (ans < m)
                return ans;
        }
    }
    return -1;
}

    In this code, we used map from C++ STL to store the values of f1(i)f1(i). Internally, map uses red-black-tree to store values. This code is a little bit slower than if we uses array and binary search for f1f1, but is much easier to write.

    Another thing to note is that, if there are multiple values of pp that has same value of f1f1, we only store one such value. This works in this case because we only want to return one possible solution. If we need to return all possible solutions, we need to change map<int,int> to, say, map<int, vector<int> >. And we also need to change the second step accordingly.

    Improved implementation

    A possible improvement is to get rid of binary exponentiation in the second phase of the algorithm. This can be done by keeping a variable that multiplies by aa each time we increase qq. With this change, the complexity of the algorithm is still the same, but now the log part is only for mapmap. Instead of mapmap, we can also use hash table (unordered_map in GNU C++) which has complexity O(1)O(1) for inserting and searching. And when the value of mm is small enough, we can also get rid of mapmap, and use a regular array to store and lookup values of f1f1.

    int solve (int a, int b, int m) {
    int n = (int) sqrt (m + .0) + 1;

    int an = 1;
    for (int i=0; i<n; ++i)
        an = (an * a) % m;

    map<int,int> vals;
    for (int i=1, cur=an; i<=n; ++i) {
        if (!vals.count(cur))
            vals[cur] = i;
        cur = (cur * an) % m;
    }

    for (int i=0, cur=b; i<=n; ++i) {
        if (vals.count(cur)) {
            int ans = vals[cur] * n - i;
            if (ans < m)
                return ans;
        }
        cur = (cur * a) % m;
    }
    return -1;
}
